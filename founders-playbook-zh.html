第一章

创业生命周期，为 2026 年重启

AI 正在重塑创业公司的构建方式。今天，从未写过一行代码的创始人正在交付生产级应用，而"精益的 10 人独角兽"也已经从一个不被看好的草根故事，变成了一个深思熟虑的行动计划。

在 2026 年，AI 能够编写生产代码、开展市场调研、综合分析竞争格局、起草投资材料，并自动化运营工作流。通过消除曾经陡峭的学习曲线——即便是经验丰富的技术型创始人，在整合实现想法所需的工具、平台和系统时也曾面对这些曲线——AI 最重要的作用,是拉平了"谁能创办公司或打造产品"这一问题上的竞争环境。

在 2026 年，一个好点子能带创始人走得比以往任何时候都远。智能体编程（agentic coding）把过去需要一整支工程师团队完成的工作，压缩成了一位创始人自己就能交付的成果。

传统的创业增长曲线假设，从想法到规模化的路径是：验证 → 融资 → 招聘 → 构建 → 再融资 → 增长 → 招更多人 → 重复。如今，AI 已经抹去了那个预期——即创业生命周期的每个新阶段都需要更大的团队、不同的技能组合和一轮新的融资。

本手册根据这些新现实，重新绘制了创业旅程的四个核心阶段（想法、MVP、发布、规模化）。我们考察当 AI 成为你技术与组织发展的核心时，每个阶段呈现出什么样貌，每个阶段适合的工具是什么，以及使用这些工具的创始人如何压缩时间线。如果你已准备好绘制从想法到退出之间的最短路径，请继续读下去。

第二章

"创始人"的定义正在改变

过去，创始人由他们"能做什么"来定义：技术型创始人写代码，非技术型创始人负责业务运营和谈成交易。但 2026 年创始人可用的模型、系统和 AI 智能体，已经溶解了"能构建的人"与"有值得构建的想法的人"之间的那堵墙。

AI 原生创业公司正在从根本上改变"创始人"的含义。如今，一个没有工程背景的人可以构建生产级软件来实现自己的想法；而一个技术娴熟、却缺乏商业知识的创始人，也能轻松产出一份市场进入策略、一个财务模型和一份高度精良的融资演示文稿。

从历史上看，创始人把大部分时间花在执行模式上：写代码、管理人员、处理日常运营工作。在 AI 原生创业公司中，创始人的角色变得更少是个人贡献者，更多是智能体的指挥者——这些专门化的 AI 助手能读取文件、运行命令、执行代码，甚至浏览网页。创始人的注意力沿着价值链向上转移，转向更高阶的工作：产生想法，并指挥执行这些想法的系统（AI 智能体、工具，以及任何存在的小团队）。

不过，AI 作为核心基础设施最具革命性的结果，是解放了具有专业领域知识的非技术型创始人。当创始人群体扩展到工程背景之外的人，你会看到由拥有截然不同人生经历的人创办的创业公司，他们解决的是传统技术创始人通道从未优先考虑（甚至从未注意到）的真实问题。

面向精益创业公司的 AI 工具能力

传统创业模式假设你需要招聘工程师来构建、招聘销售人员来卖货、招聘运营人员来管理业务。人头数被视为组织势头和产品成熟度的标志。

2026 年的早期创业公司截然不同。它们在设计上就极度精益，通常只有创始人独自一人，或一个仅几人的团队。通过将技术和组织发展都以 AI 为基础设施来构建，它们可以在扩大团队之前就达到产品验证、早期收入，甚至盈利。AI 在三个领域尤其能帮助创业公司像更大的组织那样运转：研究、智能体编程，以及自动化关键业务运营的工作流。

对话式智能与研究

设想：每个领域的随叫随到的专家

想想创始人在第一年里需要知道、却几乎肯定一开始并不知道的所有事情：我如何设置薪资发放？我如何规划产品开发冲刺？我如何起草一份严谨的投资备忘录？

过去，这类早期创业问题的答案都一样——"找一个懂的人"。对于一位自筹资金或处于种子前阶段的创始人来说，这可能消耗掉本该用于构建的时间去做知识收集，或者可能要烧掉一大块早期资本去请顾问。如今，他们拥有了一位横跨所有可想象领域的随叫随到的专家。

  • 深度研究：竞争分析、市场规模测算、财务建模
  • 文档起草：融资演示文稿、案例研究、投资备忘录、产品需求文档（PRD）
  • 战略思考伙伴：唱反调分析、事前验尸（pre-mortem）、情景规划、路线图优化

智能体编程

设想：那位永远在线、从不被阻塞的工程师

过去，构建软件需要一位技术联合创始人、一家外包开发公司，或者一段足够长的资金跑道，让你在写出一行生产代码之前先招好一支工程团队。

智能体编程工具如今让每一位有抱负的创始人都能用通俗语言描述自己想构建什么，并指挥 AI 以一整支工程团队的速度和规模来生成、测试、调试和重构生产级代码库。从"我有一个想法"到"我有一个产品"的时间线被压缩了。而创始人的角色现在聚焦于构建什么以及为什么，AI 则负责实际构建那些为真实用户准备的真实基础设施。

工作流自动化

设想：随需而动的自动化运营团队

即使一位创始人能像顾问那样研究、像工程团队那样构建，依然有一整类工作——超出战略规划或产品开发之外——仍然必须完成。排期、更新 CRM、拉取周报、保持文档最新、发布内容、跟踪合规要求、管理公司所依赖的各种工具和系统之间的连接组织，这些都得有人做。在精益创业公司里，这份负担主要落在创始人身上——而它对那些本应投向更高阶决策的时间和注意力，是一笔沉重的税。

用 AI 工具进行工作流自动化，能卸下这笔税。可以把循环往复的运营任务配置成自动发生：当一笔交易推进时 CRM 自动更新，周报自动汇编，产品文档与产品变更同步更新。而且，关键在于，Claude Cowork 能与创业公司赖以运转的相互关联的系统集成——你的项目管理工具、你的沟通栈、你的数据源——而无需有人去构建和维护这些集成。在"第零天"创业公司里，那个人几乎总是创始人。

时机与编排就是一切

有效驾驭 AI 的研究、自动化和智能体编程能力的创始人，能够构建出一家运营杠杆远超其人头数所暗示的创业公司。他们也得以把大部分时间和带宽，投入到真正重要的工作上。

这项工作不会自动驾驶；编排这些 AI 工具的创始人需要知道如何（以及何时）运用它们。本手册余下的部分，正是致力于探索创始人在沿着 AI 原生创业路径前行时将遇到的目标与挑战，以及如何在旅程的每个阶段有效地运用 AI 工具。

第三章

想法阶段

每一位创业创始人都从同一个地方出发：一个他们无法停止思考的问题。这是创业旅程中想法与现实相遇的阶段：2026 年的创业成功要求一种纪律——在证据足以支撑之前，不去构建。这一阶段的工作是研究、客户探索、竞争分析，以及对反面证据的诚实评估，所有这些都发生在你请 Claude Code 生成第一行生产代码之前。

想法阶段的目标

在想法阶段，创始人的主要目标是以研究为导向的验证：在投入资源构建之前，汇集扎实的证据，证明一个真实的问题确实存在（并且你提出的解决方案能有效应对它）。

实际操作上，想法阶段是创始人必须大致按以下顺序回答的一系列问题：

  • 这个问题是否真实、具体，且足够频繁，值得围绕它来构建？
  • 究竟是谁拥有这个问题，这群人是否构成一个市场？
  • 是否有其他人在解决它？如果有，他们怎么做，做得多好？
  • 一个解决方案究竟需要做什么才能解决这个问题，而我的想法做到了吗？

这些探询的结果加总起来，回答一个单一的、终极的问题：这值得构建吗？

这意味着在行动之前先变得具体。"人们在报销上很头疼"是一个观察。"中型企业的财务经理每周花四个多小时来核对提交的报销，因为他们现有的工具无法与会计软件集成"——这才是一个可检验的假设。

想法阶段的退出标准

想法阶段的退出条件是找到问题—解决方案契合（problem-solution fit）。你已经建立起定性证据——主要来自真实的人际对话——证明在你着手构建解决方案之前，你确实是在为真实的人解决一个真实的问题。

当你能对以下三点都回答"是"时，你就准备好离开想法阶段了：

  • 问题是否真实且具体？要在此肯定地回答，需要你能精确说出谁经历这个问题、他们多久遇到一次、它对他们的影响有多严重，以及他们目前对此采取什么应对方式。
  • 你的解决方案是否针对真正的问题？不是你最初假设的那个问题，而是验证过程所揭示的那个问题。有时两者相同，但并非总是如此。
  • 你是否有足够的信号来证明构建的合理性？在这个阶段你永远不会有确定性，而等待确定性本身就是一种失败模式；但你需要足够的定性证据，让"投入做 MVP"成为一个经过推理的决定，而不是一种信仰之举。

想法阶段的挑战

想法阶段是你创业旅程中最重要的工作发生的地方，因为它也是最具决定性的错误被犯下的地方：现在把某件事搞错，很快就能让你这家初生的企业彻底偏离轨道。不过，构思阶段的大多数挑战都涉及"行动速度超过你的理解所能支撑的程度"，因此，那些以审慎和深思熟虑前行的创始人，将体验到稳步的进展。

挑战
把"构建"误认为"验证"

当技术障碍被移除时，一位满怀热情的创始人有可能跳过创业旅程中最重要的工作：验证他们的想法是否真的是人们需要并会使用的解决方案。

即便在当前的智能体编程时代之前，就有 42% 的创业公司因为构建了无人想要的东西而失败。而如今，像 Claude Code 这样的智能体编程方案,已经极大地压缩了"我有一个想法"和"我有一个产品"之间的距离——那个失败率只会继续攀升。

虽然现在是成为一位拥有绝妙好点子的创始人的最好时代，但快速、轻松地搭出一个看起来像产品的原型，反直觉地，也为 AI 原生创业公司带来了一种真正危险的存亡风险。

直到不久之前，构建都还需要真实的开发时间和预算，哪怕拼出一个基本原型，通常也要花上几个月。然而如今技术开发的门槛在很大程度上已经消失，AI 让创始人极其容易地直接跳进构建，而不先在真实世界中验证其效用。

达到问题—解决方案契合，需要先验证假设、再构建；但许多初次创业者（甚至有经验的创始人）错误地相信 AI 让这一要求得以短路，从而把流程变成了：有一个想法 → 立刻构建一个原型 → 把原型的存在当作验证。原型变成了一个"假设一开始就对"的理由，却从未真正检验它是否成立。

一个能运行的原型，很容易被误认为是你正在解决真实问题的确凿证据，但它不是。你的原型其实是一个有用的压力测试道具，用于和潜在用户的对话。这些对话本身,才是真正的证据。

挑战
过早规模化

当构建变得毫不费力且即时，你可能让执行的规模化远远跑在业务需求之前。

过早规模化意味着在你真正验证某条产品路径是否值得投入之前，就承诺走上那条路。

这一直是创业杀手，但 AI 让创始人陷入过早规模化陷阱而不自知变得容易得多。智能体编程助手如此强大，以至于很容易在从未有意识地决定偏离航向的情况下，就让执行的规模化远远跑在验证问题—解决方案契合之前。它会围绕一个根本上有缺陷的前提去生成、测试、调试和重构代码库，所带的热情与它对待一个绝妙想法时一模一样。系统中的智慧来自你。这个阶段的首要指令，是让你的意义建构（sense-making）始终领先于你的构建，尤其是当构建如此迅速、感觉如此轻松的时候。

挑战
客观性的丧失

向一个 AI 工具索要支持你已经相信之事的证据，它就会找到。如今，确认偏误配上了一台研究引擎。

确认偏误一直是创业中的职业危害：创始人天性就对自己的想法充满热情。如今，AI 工具给了确认偏误一次重大的能力增强。让 AI 验证你的创业想法，它就会找到支持的证据；让它测算你的潜在市场规模，它就会找到那个让你的 TAM 看起来值得投资的数字。

AI 跟随你的指引，这意味着一个不去问尖锐问题的创始人，现在能比以往任何时候都更快地为一个糟糕的想法构建出一个精致、看起来研究充分的论证，同时还感觉自己确实是在做尽职调查。解药是同一个工具，只是指向相反方向：AI 会像验证一个想法那样彻底地对它进行压力测试。当研究和结构化的对抗性思考浮现出"你的想法需要修正"的证据时，这就是转向（pivot）的信号。

定义并压力测试问题假设

练习

与 Claude 一起打磨你的问题陈述，直到它成为一个可检验的假设。例如，"合同审查耗时太长"并不真正可检验。但"中型企业的内部法务团队在每个合同审查周期上花费 3 天以上，因为修订意见是通过邮件串而非单一的版本受控文档来管理的"——这就非常可检验。

注意把 Claude 用作结构化的唱反调者，是 AI 创业生命周期每个阶段的一项核心用例。

市场研究与绘制竞争格局

掂量你的竞争对手。有一种创业公司特有的现象叫做"竞争对手忽视"：过于强烈地聚焦于自己的愿景和执行，以至于系统性地低估同一领域中其他人正在做的事。幸运的是，AI 提供了解药：请 Claude 提出最有说服力的论证，说明为什么这个解决方案领域中的某个竞争对手会成功而你不会。Claude 能分析为什么他们的方法实际上更好、为什么客户会选择他们、为什么你潜在的差异化优势可能并不如你以为的那样难以撼动。

练习

请 Claude 按层级绘制你的竞争格局：直接竞争者、间接竞争者、潜在收购方，以及可能进入你领域的相邻玩家。然后请它论证为什么每一层都对你的成功构成真正的威胁——不是那种最容易被驳回的威胁版本。

市场研究。Claude Code 能综合公开可得的客户反馈，浮现出反复出现的抱怨和未被满足的需求。额外收获：这样做本质上是对你竞争对手的客户做的免费定性研究。

练习

指挥 Claude Cowork 综合你关键来源中的竞品评价，识别出现有方案尚未解决的首要抱怨。如果你的假设解决了其中一项或多项，那是问题—解决方案契合的强力证据。如果没有，那也值得知道。

Claude Cowork 还能从密集的行业报告、分析师文件和市场研究文档中提取相关信息和数字；接着，这些干净、综合过的输入便成为 Claude 分析工作的理想上下文。

练习

基于公开数据构建 TAM/SAM/SOM 模型，并对其背后的假设进行压力测试。判断市场是在扩张、整合还是已经成熟；这一背景会影响你对时机和差异化的思考。绘制买家格局：谁掌握预算、谁影响决策，以及这两者是否是同一个人。

趋势分析。最后，用 Claude 来倾听那些告诉你"是否在正确时机进入"的早期指标。追踪那些关于你问题的对话已经在发生的 subreddit 和 LinkedIn 群组，以及用户描述自己困扰时所用的确切语言。请 Claude 识别出类似问题曾被解决的相似市场，并提取出哪些做法奏效、哪些没有。浮现出可能加速或威胁这个机会的监管、技术或人口趋势。

练习

请 Claude 识别出三个可能在未来两年内显著影响你市场的外部趋势——监管、技术或人口方面的——并评估其中每一个对你的具体假设而言是顺风还是逆风。

注意本节中的市场研究和竞争格局绘制工作不是一次性的练习。你将在 MVP 和发布阶段继续有所发现并演进你的思考，因此每当你的假设演变时，重做这些练习都很重要。

规划并设计客户探索

你通过与产品的潜在用户交谈所学到的东西的质量，由以下两点决定：（1）你所提问题的质量；（2）你是否把这些问题问给了正确的人。Claude 在进行客户探索方面尤其有帮助，包括找谁谈、问什么，以及如何理解你听到的内容。

找谁谈。一份精确的目标画像，远比一长串联系人名单有价值得多，它包括最可能强烈经历这个问题的具体职位头衔、公司类型、团队结构和资历层级。从那里出发，识别出这些人实际上可以在哪里被触达——他们聚集的社区、活动、LinkedIn 群组和 Slack 工作空间——并基于他们离问题有多近，构建一个"先联系谁"的优先级框架。

问什么。定义好目标后，用 Claude 来构建访谈框架本身：正确的问题，以正确的顺序，结构化地设计成浮现出人们实际做什么，而不是他们认为自己会做什么。新手创始人的一个错误是问一个关于未来的、笼统的开放式问题（"你会用类似这样的东西吗？"），而不是具体地询问相关的过去（"跟我讲讲你上一次处理这个问题的情形。"）。Claude 也能标出你的草拟问题中哪些在引导受访者、哪些太宽泛，或者哪些可能产生噪音而非信号。Claude 还能帮你设计追问问题，用以探查回避，或对重要问题的含糊回答深入挖掘。如果你的假设涉及不止一个人物角色，Claude 还能为每个角色设计不同的问题集。一位财务经理和一位 CFO 与同一个问题的关系是不同的，单一的访谈框架会抹平这种区别。

练习

先亲手起草你的访谈问题，再请 Claude 审核它们。具体地请它标出任何具有引导性、面向未来、太宽泛，或可能产生社会期许式回答而非诚实回答的问题。然后请它为访谈中最可能产生回避的两三个时刻，各建议一个追问。

访谈后分析。每次对话之后，用 Claude 来复盘：把你的笔记喂给它，请它识别出什么证实了你的假设、什么挑战了它，以及什么是真正出人意料的。一旦你积累了一批访谈，把你完整的访谈笔记集放进 Claude Cowork，浮现出反复出现的主题、矛盾，以及双向上最强的信号。然后把那份综合输出带回给 Claude，请它标出你自己对数据的解读可能在哪里是在向"你想听到的"做模式匹配，而不是"实际存在的"。

练习

每做完五次访谈，指挥 Claude Cowork 综合你的笔记并产出两份清单：支持你假设的证据，以及挑战你假设的证据。如果第一份清单明显长于第二份，请 Claude 判断这种不对称反映的是数据中实际存在的东西——还是你曾希望找到的东西。

客户触达与排期。用 Claude Cowork 来自动化围绕构建联系人名单、开展触达和安排用户访谈的运营负担。Claude Cowork 能使用你与 Claude 一起定义的目标画像（包括职位头衔、公司类型和资历层级）来研究并编制一份结构化的潜在客户名单和经核实的联系信息。接着它会大规模起草个性化的触达邮件，根据每个人的角色和背景量身定制。随着回复进来，它通过 MCP 连接 Gmail 和 Google 日历来管理邮件串、处理排期请求并把访谈排进日历。工作流继续：Claude Cowork 按既定节奏生成跟进草稿（例如对七天未回复的联系人发出第七天跟进），并在每一步完成时更新你的追踪表，让你始终知道每个潜在客户处在管道的哪个位置。

练习

把你经过验证的访谈目标画像交给 Claude Cowork，请它构建一份潜在客户名单、起草一个个性化的触达序列，并设置一张带有"触达状态、跟进节奏、访谈完成"列的追踪表。然后让它去运行这项协调工作，而你专注于为对话本身做准备。

设计你的最终解决方案概念

你已经完成了验证工作：问题是真实的，你知道谁拥有它，并且你有一个被证据支持的解决方案概念。用 Claude 从各个角度来发展并挑战你的解决方案概念：缺口在哪里？存在哪些替代方案？要让这个解决方案在规模上奏效，需要什么前提为真？这是一个重要的现实检查点：这个设计真的针对验证过程所揭示的那个问题吗，而不是你最初进来时假设的那个问题？

练习

把你的解决方案概念呈现给 Claude，请它识别出你的设计最重度依赖的三个假设。然后问：要让每个假设成立需要什么为真，以及如果其中任何一个不成立会有什么后果。

用 Claude Code 构建一个轻量原型

现在到了有趣的部分：有了一个经过验证的假设和一个经过压力测试的解决方案概念，你终于准备好构建某样东西了。

这是想法阶段中 Claude Code 登场的时刻。即使你一路上一直在小修小改，现在才是你生成正式轻量原型的节点：把你的想法摆到一个真实的人面前并获得真实反应所需的最小表面积。

你（还）不是在构建一个真实世界的产品；你是在构建一个你想法的功能性样本，用于客户和投资者对话。真实用户对一个他们能够实际触碰的东西的反应，会告诉你十几次问题—解决方案探索访谈都告诉不了你的东西。之前，你是在确立"你正在解决的问题是真实的"；现在，你是在请潜在用户与所提出的解决方案互动。

练习

定义你的解决方案所依赖的那个单一核心交互。指挥 Claude Code 只构建那一个。当你拥有它时，把它摆到五位来自你已验证目标画像的人面前，请他们试用。你在那五次对话中学到的东西，将决定你是继续构建，还是回到画板重来。

抵达想法阶段的终点，是在 AI 创业竞赛中向前的一次巨大飞跃，因为现在你不再是在押注一个直觉；你是在对照证据来执行。接下来是 MVP 阶段，创始人的引导性问题从"这值得构建吗？"变成"我们究竟应该先构建什么？"，而 AI 的主要角色也从研究伙伴转变为施工队。

第四章

MVP 阶段

不少创始人把 MVP 阶段当作一个施工阶段，但 MVP 阶段从根本上仍然是一项证据收集练习。区别在于，你现在收集的是关于解决方案的证据，而不是关于问题空间的证据；具体来说，就是一群真实、可识别的人是否觉得它有足够价值，值得使用它、回访它、为它付费，和/或把它告诉别人。

MVP 阶段的目标

作为 AI 原生创业公司的创始人，你的目标是把一个经过验证的问题，转化为一个真实用户会真正使用的可用产品。这不是带有每一项路线图功能的完整版本，而是你想法的最小、最聚焦的迭代——它把一个真实的解决方案摆到真实用户面前，并产生关于产品—市场契合的真实证据。

与此同时，你现在如何构建，决定了以后什么是可能的。这意味着 MVP 阶段有一个同样重要的第二目标：快速行动，同时不积累那种会复利累加的技术债——这种债务会在真实用户大量到来的那一刻就来纠缠你。

最后，从第一天起就投资于持久的上下文，正是让 AI 保持为力量倍增器、而非熵的来源的关键。在 AI 原生创业公司中，你的代码库是你与 AI 一次又一次会话中协作的对象，这使得"可读性/可理解性"成为基础。跳过规格说明、架构决策和上下文文件（如 CLAUDE.md）的创始人，会撞上一堵可预见的墙：每个新会话都需要重新解释代码库，而 AI 生成的改动会偏离最初的愿景。

MVP 阶段的退出标准

MVP 阶段的退出条件是产品—市场契合的真实证据：证明一个具体的、可识别的用户群体已经发现产品有足够价值，值得回访它（留存）、为它付费（收入），或把它告诉别人（推荐）。

MVP 阶段的挑战

在 MVP 阶段，创始人的首要指令是速度与判断力。这里的挑战集中在：你能否足够快地，以正确的方式，构建出正确的东西，而不走那些日后会让你付出代价的捷径。

挑战
智能体技术债

因为 AI 本质上移除了每一个曾经控制"什么能进入生产"的天然瓶颈，速度是有保障的。但当速度成为创始人在 MVP 构建中纳入考量的唯一变量时，他们就有积累自己将难以偿还的技术债的风险。

在 MVP 阶段，有些技术债是恰当的——前提是理解它必须在规模化之前被管理。它逐渐积累，可以随时间清理或在一个专门的冲刺中清除。然而，AI 技术债是复利累加的。如果没有把规格说明和架构约束写在 AI 能读到的某处，每个会话都会从头重新推导基础性决策，而那些决策会漂移。你最终会得到一个背后没有连贯心智模型的代码库——不是因为任何单个部分糟糕，而是因为这些部分从来就不是为彼此契合而设计的。那是一个真实的问题，而且它确实倾向于很晚才浮现。

挑战
误信虚假的产品—市场契合

AI 工具能生成令人印象深刻的早期数字，但这些并不保证市场需要你的产品。

早期势头是创始人能拥有的、心理上最强大的体验之一。在数周或数月的验证工作和谨慎、有纪律的构建之后，交付一个产品感觉像是对"你一直以来都是对的"的确认。

智能体编程工具能帮你比以往任何时候都更快地抵达这一刻，但早期牵引力并不等同于产品—市场契合。发布的能量产生于短暂的力量，比如你创始人的朋友、你投资人其他被投公司里的潜在买家，或者一条推动流量激增的 Hacker News 头条。不幸的是，这些都不能可靠地预测第六周或第十二周——当那股初始推力消退之后——会发生什么。

挑战
零摩擦的范围蔓延

当构建感觉毫不费力、几乎免费时，总有一个更酷的功能可以加，或者一个更多的边缘情况可以处理。这种范围蔓延弊大于利。

范围蔓延一直是创业风险。如今的不同在于，对抗它的传统强制约束——工程时间的真实成本——在添加一个功能只需一个下午而非一个冲刺时，已不再以同样的方式存在。

这里的挑战是，每一项单独的添加都站得住脚。产品当然应该处理那个边缘情况；用户当然会想要那个工作流。它们在当下并不感觉像范围蔓延，因为用智能体编程构建每一个都只需如此之少的努力；但随着你的产品蔓延越出其原本边界，你将面临失去方向和势头的风险。

解药是一份在构建开始之前就创建好的、书面的范围定义，它描述产品做什么、它刻意不做什么，以及什么样的来自真实用户的具体证据才能证明添加新东西是合理的。这把决策点从"我们应该构建这个吗？"移到"是否已有相当数量的用户告诉我们，没有这个他们就无法从产品中获得价值？"

挑战
因经验不足而不安全

使用 AI 工具在没有先理解基本安全原则的情况下匆忙把应用推向市场的创始人，最终会让他们的用户暴露于本可预防的风险之中。

残酷的事实是，智能体编程工具生成的是能运行的代码，而不是本质上安全的代码。功能性代码很容易，因为要么功能能用，要么不能用。安全漏洞在被利用之前是不可见的，这意味着没有天然的反馈回路来提醒一位初次创业的创始人有什么地方出了错。然而，把一个活的 MVP 交付给真实用户，意味着真实的数据、真实的暴露，以及如果出了问题就有真实的后果。

轻视安全对 AI 原生项目来说并不新鲜。每个时代的自筹资金创业公司,往往都把安全考量推迟到构建后期，有时一直等到生产发布的边缘。在任何用户接触你的应用或方案之前进行一次安全审查，是把一个最小可行产品放到世界上的、负责任的最低门槛。

Claude 如何帮助 MVP 阶段的创始人

在构建之前定义你的架构

在 Claude Code 写下一行生产代码之前，用 Claude 来定义并记录将统辖这一阶段所有构建的架构决策：要遵循的模式、要避免的依赖、正在做出的权衡及其原因。这份输出将充当一份聚焦的架构上下文文档，并确立 Claude Code 将在其中运作的护栏。

没有这份上下文，每个会话都从零开始，Claude Code 被迫推断自己的结构性假设。让 Claude Code 在没有护栏的情况下构建，会产出一个功能可用但结构上不连贯的代码库；而迭代和扩展不连贯的代码库，归根结底是对时间和 token 的浪费。迟早会有一个节点，代码不可避免地崩塌，迫使你从头重建。

练习

在打开 Claude Code 之前，先打开 Claude 并描述你正在构建什么：它解决的核心问题、它服务的用户，以及你在接下来六个月里现实预期的规模。请它帮你定义应当统辖你 MVP 构建的架构原则、考虑到你的约束应当避免的依赖，以及你在这个阶段有意识地接受的权衡。接着，把这份输出保存为 CLAUDE.md markdown 文件。这是你的架构上下文文档：你构建的第一个产物，也是之后每个会话所依赖的那一个。CLAUDE.md 文件充当 Claude Code 的项目级指令，提供项目特定的上下文和指令，当 Agent SDK 在某个目录中运行时会被自动读取。从功能上说，它们是你项目的持久"记忆"。

定义并强制执行你的 MVP 范围

无摩擦的范围蔓延是 AI 时代 MVP 的标志性失败模式之一。正如你定义并记录了产品的应用架构，你同样需要在第一个功能被构建之前定义你 MVP 的范围。

Claude 能帮你创建一份范围文档，描述你的 MVP 产品做什么、它刻意不做什么，以及功能修订标准：什么样的来自真实用户的具体证据，才能在这个节点证明添加新东西是合理的。当新的功能想法浮现时——它们必然会——你用 Claude 来压力测试它究竟是来自用户的真实信号，还是包装成产品思考的创始人热情。

用 Claude Code 构建你的 MVP

一旦架构和范围被定义，Claude Code 就成为主要的 MVP 构建工具。用它来生成、测试、调试和迭代你的代码库，但把每个会话都当作对你已经做出的产品决策的执行，而不是抛出一些新决策的机会。

每个 Claude Code 会话开始时，（1）重温你的范围文档，并（2）向模型提供你的 CLAUDE.md 架构上下文文档。每个会话结束时，用该会话浮现的任何决策来更新它。目标是一个其结构你能解释清楚的代码库，而不仅仅是一个能运行的代码库。

练习

为你的 Claude Code 工作创建一个简单的会话模板，包含架构上下文文档、本次会话的具体任务，以及任何需要遵守的约束或模式。每个会话结束时，向上下文文档添加一条简短的日志条目，详述构建了什么、做出了什么决策、本次会话引入了什么假设。每个会话花五分钟做文档记录，是对抗"复利累加成无法管理的代码库"的架构漂移的廉价保险。

在任何用户接触它之前进行安全审查

作为 AI 原生创业公司的创始人，你的责任是知道你代码库里有什么、理解任何潜在的暴露途径，并且不把明显的漏洞交付给那些信任你、把数据托付给你的真实用户。

Claude 能对 AI 生成的代码做一次有用的初步安全审查，并能帮助识别常见漏洞。把它纳入交付前的循环是一个好习惯。然而，它不是安全工具的替代品，在更高风险下，也不是人类审查者的替代品——把它当作替代品的创始人，正是那些最终出现在数据泄露故事里的人。Claude Code Security 走得更远：它扫描代码库以查找安全漏洞，并建议有针对性的补丁供人类审查，浮现出传统方法可能错过的问题。

注意在本电子书出版之时，Claude Code Security 是一个有限的 beta 版本，因此在把它纳入你的工作流之前，请核查当前可用性。

练习

在部署给任何真实用户之前，用一份具体的简报把你的核心应用代码过一遍 Claude：审查身份验证和会话处理、API 响应中的数据暴露、输入验证和注入风险，以及有已知漏洞的依赖。认真对待每一项发现，评估它是否需要修复，对任何触及身份验证、密钥或数据处理的部分都进行人类审查。

在发布之前构建你的度量框架

那些把早期牵引力误认为产品—市场契合的创始人，通常正是那些在发布之后才开始追踪数据的人——用的是为评估"什么在奏效"而非浮现"什么没在奏效"所选的指标。解药是在第一个用户出现之前就确立你的度量框架。

用 Claude 来定义哪些指标对你的具体产品重要、基准是什么，以及数据中什么样的模式才构成真正的产品—市场契合而非讨喜的噪音。具体来说：在发布你的 MVP 之前，设定好你的留存基准、你的激活标准，以及你的第 7 天和第 30 天目标。接着，定义对你的具体产品而言一个假阳性长什么样：例如注册却不激活、有收入却无留存，或者初始热情却无重复使用。当数据到来时，请 Claude 对你自己的牵引力做对抗性论证：对这些数字，一个怀疑者会怎么说？

管理探索与用户反馈的运营事务

一旦真实用户进入产品，运营层就会快速扩张。Claude Cowork 处理那些重要但繁琐的工作，比如构建和维护用户联系人名单、运行触达序列、安排反馈会谈、分类整理 bug 报告，以及追踪迭代周期。在想法阶段管理探索事务的那些相同的 MCP 集成，在这里同样适用。

在收集回路中保留一个人类，用于对用户反馈做细致入微的探究。例如一位用户说"这很棒，但我希望它还能……"，这需要解读：这是核心需求还是锦上添花？是这个客户特有的，还是代表了一个细分群体？缺失的功能是真正的问题，还是上游的入门引导里有什么问题？没有工具能回答这些问题。

练习

配置 Claude Cowork 来运行你的 MVP 阶段反馈回路：向你的早期用户名单起草触达、安排反馈会谈、为 bug 报告和功能请求设计结构化的接收流程，并写出一份关于进来内容的每周综述。先自己审阅这份综述；之后，你可以请 Claude 分析这些信息，捕捉你可能忽略的任何重要要点。

朝着证据迭代，而非朝着完整性

MVP 阶段在你拥有产品—市场契合的真实证据时结束，无论产品感觉多么"完成"。宣布你已经达到产品—市场契合、现在准备从 MVP 阶段进入发布阶段，归根结底是一项结合了创始人直觉与所收集证据的判断练习。不过，有一些有用的试金石：

  • Sean Ellis 测试：问你的活跃用户："如果你再也不能使用这个产品，你会作何感受？"如果超过 40% 的人回答"非常失望"，那是一个有意义的 PMF 指标。
  • 努力测试：在产品—市场契合之前，留存需要持续的干预，包括频繁的触达、激励、个人跟进，以及为保持用户参与而付出的英雄式创始人能量。在产品—市场契合之后，产品开始自己做那项工作。当事情开始"拉"而不是"推"时，那种努力上的转变,是某种真实的东西已经改变的最清晰信号之一。

归根结底，没有任何单一的数据点能确认产品—市场契合，因为它是一种必须跨多个迭代周期都成立的模式，你才能确定地宣称它。

当证据要求时，转向

如果，即便投入了所有这些工作，你就是似乎无法抵达产品—市场契合，怎么办？你的结果不能证实你出发时的方向，这并非失败，而是系统在起作用：MVP 阶段的设计初衷，正是在你对一个错误答案过度投入之前浮现出这一信息。

当数据不支持你当前的产品时，用 Claude 来梳理那份数据在告诉你什么。

  • 探索其他客户细分。也许那些没有转化的用户一开始就不是正确的目标。正确的受众往往已经在你的数据里，只是被低估了。
  • 调整你产品的价值主张。也许你的受众是对的，但你的 MVP 就是没有引起用户共鸣。对入门引导、信息传达或核心功能侧重点的一次调整，有可能在不改变你已构建之物的情况下修复这一点。

对"这种脱节可能深到需要一次更根本的改变"的可能性保持开放。

练习

如果你已经完成了三个或更多迭代周期，却没有朝你的产品—市场契合基准有意义地移动，在决定下一步做什么之前，用 Claude 来运行一次诊断。把你的留存数据、用户反馈和最初的问题假设喂给它，并问它三个问题：（1）这份数据里是否有一个细分群体的反应与其余群体不同？（2）"设计价值"与"体验价值"之间的差距，是一个定位问题还是一个产品问题？（3）要让当前产品找到真正的 PMF 需要什么为真，而鉴于你所看到的，那个情景是否现实？让答案来决定你是调整、转向，还是回到想法阶段。

第五章

发布阶段

如果 MVP 阶段是关于证明你的产品配得上存在，那么发布阶段是关于证明你的业务配得上增长。

发布阶段的目标

在发布阶段，创业创始人必须把早期牵引力转化为一台可重复、可持续的增长引擎。除了让你的产品达到生产就绪，你还必须在其下方加固基础设施，同时围绕你的产品构建一家真正的公司。

创业公司在想法和 MVP 阶段天然以创始人为中心，因为你需要完整的态势感知和紧密的反馈回路。然而现在，仍然试图亲自抓住每一根线头的创始人，会变成发布阶段的瓶颈。目标不是把你自己从公司中移除，而是构建运营系统，把你的注意力解放出来,用于只有创始人才能做的决策。

发布阶段的退出标准

发布阶段的退出条件有三个要素：

  • 增长是可重复的、由渠道驱动的。你不只是在留住用户，你是在通过具体渠道、以被理解的单位经济模型可预测地获取他们：CAC、LTV 和回收周期都是你知道并能为之辩护的数字。
  • 产品能承受生产负载。基础设施已加固，安全与合规已就位，可靠性在真实的生产条件下成立（不只是你测试过的条件）。
  • 运营在没有创始人瓶颈的情况下运转。流程已存在，自动化已到位。你不再是亲自处理支持、分类、冲刺规划或报告的那个人。

发布阶段的挑战

找到产品—市场契合是早期创业生命周期中最难的问题。如今，创始人的挑战变成了守住它。发布阶段是这样一个地方：找到了真实产品牵引力的公司，如果围绕和支持产品的组织跟不上，仍然可能分崩离析。以下是需要警惕的失败模式。

挑战
技术债到期

为速度和验证而构建的 MVP 代码库运行得足够好，证明了产品能用，但生产流量、新功能和不断增长的复杂度，现在正在暴露那些捷径。

在 MVP 阶段，积累一些技术债是为速度而做的合理权衡。在发布阶段，那笔债开始累积利息，而它越久不被处理，修复起来就越昂贵。解决方案包括：一次系统性的架构审计以识别结构性弱点、有针对性的重构以处理其中最糟糕的部分，以及测试覆盖率的有意义扩展，让下一轮功能工作不会重新引入相同的问题。

挑战
创始人成为瓶颈

在 MVP 阶段，创始人身处每一个回路是一项资产。在发布阶段，随着支持量增长、产品决策堆积、运营复杂度倍增，同样的本能变成了约束。

从"做工作"到"设计做工作的系统"的转变，是创业生命周期中最艰难的转变之一。因为很少有一个清晰的时刻标志它的发生，风险在于完全错过它，停留在构建者模式，而组织在你周围停滞。表明这正在发生的迹象包括：本该花一小时的决策现在要花一周你才腾出手来处理；支持请求堆积，因为只有你知道答案；以及运营任务只有在你亲自记得时才会发生。补救办法是对你亲自处理的一切——从最微小的任务到最高风险的决策——进行一次彻底审计，以识别什么可以系统化、什么可以委派，以及什么才真正值得创始人的时间和注意力。

挑战
安全与合规不再可推迟

在 MVP 阶段保持安全与合规措施简单是可以的，但现在，有了真实用户、真实数据，以及桌上可能摆着的企业合同，它就变成了一项责任风险。

在 MVP 阶段，只有少数 beta 用户、生产环境中没有敏感数据时，安全漏洞是理论上的风险。然而，当你的产品带着依赖它的真实用户进入生产的那一刻，假设就变成了非常真实的暴露风险。此外，不适用于原型的合规要求，在你处理客户数据、处理支付，或向受监管行业销售的那一刻，绝对适用。补救办法是在生产规模到来之前——而不是之后——进行一次系统性的安全与合规审查，并把浮现的一切都当作必需的整改——而不是一个建议——在下一波用户到来之前完成。

挑战
在你准备好之前扩张

新市场和融资机会看起来像增长机会。它们也可能是产品—市场契合走向消亡的地方。

你建立的初始牵引力是真实的，但它也特定于你的早期受众。过早扩张进一个与你最初市场有实质差异的市场，会引入你的产品并非为之设计的新用户行为、合规要求、支付基础设施和基线预期。突然之间，有太多新变量，你失去了清晰解读自己数据的能力。你也面临着在追逐一个新的、未经检验的受众时忽视你原始用户群的风险。

Claude 如何帮助发布阶段的创始人

三种形态的 Claude 在发布阶段都被充分使用，并且它们彼此支撑：每个工具产出的输出都成为另外两个的输入。结果有机地复利累加，一位同时使用全部三个工具的创始人，得到的远多于它们之和。

这正是让超精益创业模式在结构上成为可能的原因。当 Claude Code 构建产品、Claude Cowork 围绕它构建公司、Claude 帮助把产品和组织知识运营化时，一个小团队就能像一家其规模 n 倍的公司那样运转。

在技术债复利累加之前整改它

你的 MVP 代码库能用，但它也需要一次系统性的整改遍历，去寻找任何可能变成结构性责任的技术债。首先，用 Claude Code 运行一次完整的架构审计：识别代码库在哪里脆弱、哪些捷径将变得维护昂贵，以及测试覆盖率在哪里薄到下一轮功能工作会重新引入相同的问题。把 Claude Code 的审计发现反馈给 Claude，以分类和排序整改工作：什么需要在下一次发布前修复、什么可以等一个冲刺，以及鉴于你当前阶段什么代表可接受的持续债务。这也是记录你在 MVP 阶段所做架构决策的时刻（那些因为没时间写下来而一直留在你脑子里的决策）。现在把它们写进一个 CLAUDE.md，能确保每个未来的 Claude Code 会话都从"系统如何被设计以及为什么"的共同理解出发。

练习

指挥 Claude Code 审计你的 MVP 代码库，产出一份优先排序的结构性弱点、测试覆盖率缺口和重构候选项清单。然后把那份清单喂给 Claude，请它把整改工作排布到你的若干个冲刺中：你需要首先处理的任何重大问题、可以与功能开发并行处理的事项，以及可以等待的事项。

构建替代创始人注意力的系统

构建能把你的注意力解放出来、用于处理只有创始人能应对之职责的运营系统，需要确切知道你的注意力正流向何处。用 Claude Cowork 对你当前的运营负载运行一次结构化审计，记录每一项循环往复的任务、每一个落到你桌上的决策，以及每一个只因为你亲自记得才发生的工作流。然后让 Claude Cowork 把这份清单分类为：什么可以完全自动化、什么需要一个人但不一定是你，以及什么真正需要创始人判断。审计完成后，用 Claude Cowork 为自动化候选项设计工作流逻辑：什么触发每个工作流、决策规则是什么、输出长什么样，以及完成后它去向何处。

把安全与合规作为一条产品工作流

用 Claude Code 浮现出在 SOC 2、GDPR 或 HIPAA 审计中频繁出现的代码层问题，以及你目标市场要求的标准。这会同时浮现漏洞和合规缺口。把这些发现喂给 Claude，帮你优先排序整改工作，并设计企业买家在签约前会要求的控制措施、审计日志和访问管理。注意：AI 扫描是一种辅助，但不是合格合规审查的替代品。

接下来，把合规工作流构建进你的开发周期，而不是把它作为一个一次性项目来运行；合规文档需要持续维护和更新。对于接近企业合同或国际市场的创始人来说，这也是 Claude Code 安全扫描能帮你为一次独立安全评估做准备的时刻。

练习

用面向你目标市场所要求框架的 Claude Code 运行一次代码层安全审查。把输出喂给 Claude，请它产出两样东西：一份优先排序的安全整改序列，以及一份你需要产出的文档和控制措施清单，用以满足来自潜在企业买家的合规审查。

建立你一直在跳过的产品管理流程

发布阶段需要一套轻量、可重复、无需创始人干预即可触发或运行的流程。用 Claude 来设计你的产品时间线和工作周期将如何构建、在 Claude Code 触碰一个功能之前一份规格说明需要包含什么、bug 报告如何被分类和路由，以及你的每周指标报告涵盖什么、如何分发。流程设计完成后，用 Claude Cowork 来构建并运行运营层：安排冲刺仪式、把进来的 bug 报告路由到正确的地方、从你连接的数据源汇编每周指标，并维护那个让用户信号持续流入产品决策的反馈回路。

练习

请 Claude 设计一个轻量的产品管理操作系统：一个确定的冲刺节奏、一份最小规格说明模板、一棵 bug 分类决策树，以及一份从你实际数据源拉取的每周指标简报。然后设置 Claude Cowork 来实施并运行该系统的循环往复运营要素，比如排期、路由和报告汇编，让它们按计划发生而无需你。

第六章

规模化阶段

在规模化阶段，创始人的角色从构建者重新定位为面向公众的高管。产品仍然处于核心，但你个人的日常工作越来越多地是关于公司本身。你的注意力必须扩展到新的规模化阶段活动，比如分析师简报会和 IPO 路演，即便你仍力求维持那个精益的、以 AI 为中心的结构性优势。

规模化阶段的目标

扩展技术基础设施的工作一直在继续，现在又加上了扩展组织本身、并把它成熟为一门生意的工作。

在规模化阶段，你面对的是从数千用户走向数百万、从一个市场走向多个市场。在之前的每个阶段，增长都是你可以靠近用户、基于来自紧密反馈回路的数据加上一剂健康的创始人直觉来调整航向、从而摸索着前进的东西。然而现在，目标是构建由成熟组织运营所支撑的系统化增长。

对一家 AI 原生创业公司而言，你的目标应当是通过累积的深度构建一道可防御的护城河，这道护城河源自你构建进产品的专业知识、你产品与用户依赖的其他工具和平台的集成深度，以及专有的系统数据和工作流。那些一直朝一个方向、在一致的基础设施上持续构建的创始人，现在拥有了真正难以复制的东西。

在这个阶段，公众投资者、分析师、监管者、企业采购团队和收购方施加更大的压力——连同更大的怀疑——因为现在风险更高了。你的产品和组织必须经受外部审视：不只是你所构建之物的能力，还有围绕它的治理、合规姿态、财务控制和战略叙事。

规模化阶段的退出标准

规模化阶段的退出条件不再是单一里程碑，而是一个门槛事件：即便创始人越来越不直接运营日常运营，公司依然可持续。你已经展示了系统化增长；构建了满足最苛刻外部审查者的组织治理与合规基础设施；并且对"如果一个资金充裕的现有巨头今天复制了你的产品，你的用户会留下吗？"这个问题有一个扎实的答案。

实践中，这个门槛通常呈现三种形式之一：在不再需要外部资本的规模上实现可持续盈利、IPO 就绪，或被收购。三者都要求你的增长是系统化且可审计的、你的产品护城河经得起审视，以及你的组织在运营上成熟且可持续。

当这一切为真时，恭喜你：你的创业公司已经从一场赌注，变成了一门生意。

规模化阶段的挑战

挑战
委派运营层

规模化阶段的运营系统必须可靠、可持续地运转，而无需被照看。对于一位从第一天起就亲力亲为的创始人来说，那个转变既是结构性挑战，也同样是心理挑战。

你发布阶段的工作是创建这些系统；在规模化阶段，它变成了（1）把这些系统成熟到完全可信任，然后（2）真正地信任它们。这比听起来更难。即便你是一位善于委派的创始人，也并不总是显而易见该交出什么、该留在自己盘子里什么。交出太多、太快——尤其是交给 AI 自动化系统——关键决策就会在缺少只有创始人能提供的关键上下文的情况下被做出。然而抓得太久，你就会成为瓶颈。这里的根本挑战是识别那些只存在于创始人脑中或未被记录的工作流中的机构知识，然后把它编码进有文档记录、可审计且可转移的系统。

挑战
扩展技术运营

客户不再只评估你的产品；他们想知道你的组织能成为一个可依赖的基础设施伙伴。

前三个创业阶段的技术挑战集中在代码库上：在不积累技术债的情况下构建正确的解决方案，然后为真实用户加固安全与合规。抵达规模化阶段后，挑战现在变成了围绕代码库构建的一切；创建那些标志成熟度的支持基础设施、文档和可靠性保证。更大规模的客户和签署多年合同的机构买家，在签约前想要这些，签约后也会据此要求你。不过，把你带到这一步的同样的 AI 基础设施，会帮你构建带有确定响应时间的专门支持职能，以及一份新客户的工程团队真正能用的文档。

挑战
扩展组织职能

一家规模化阶段的公司通常需要诸如招聘、薪资、会计和法务运营之类的组织基础设施，无论有多少人在运营它。

在发布阶段，系统化运营意味着自动化那些消耗创始人注意力的工作流。一家规模化阶段的创业公司现在需要培育一个更宽广、在某些方面也更具决定性的运营职能阵列，比如财务报告、合规监控、合同管理和客户支持，等等。

挑战
构建一个 GTM 职能

有机增长有一个天花板，而大多数规模化阶段的创始人在不得不构建一个真正的市场进入（go-to-market）职能之前，就撞上了它。

想法、MVP 和发布阶段的增长往往源自创始人主导的销售，从一篇时机得当的 Product Hunt 帖子，到与早期客户的私人关系。然而像这样的有机增长只在一定程度内奏效，大多数创业公司在规模化阶段撞上这个极限。迹象包括用户曲线趋平、客户获取成本上升，以及一条只有当创始人亲自参与时才推进的管道。规模化阶段的增长需要构建一台专门的增长引擎，以触达你产品的新的、更宽广的受众。然而，大多数创业创始人，很可能从未不得不运营过营销、销售和分析师关系项目这类东西。一个正经的 GTM 动作不仅需要建立新的系统和流程，还需要为"你想如何谈论你的产品"创造一种品牌声音和故事。因为，在创业生命周期的这个阶段，你将需要它来触达的不只是个体新用户，还有整个目标受众,比如投资者和企业买家。幸运的是，GTM 职能不必很大就能有效，而构建了产品的同一套 AI 基础设施,可以引导把它带向市场。

Claude 如何帮助规模化阶段的创始人

早期创业阶段把 Claude 用作产品本身的基础设施：验证想法的研究伙伴、设计和构建原型的工程团队，以及让单创始人创业公司成为可能的 AI 运营层。抵达规模化阶段的 AI 原生创业公司创始人，现在可以用 Claude、Claude Code 和 Claude Cowork,以他们构建时的同样方式继续扩展。

把日常任务交接给 Claude Cowork

带着对"现在你最需要把时间和注意力投在哪里"的清醒认识开启规模化阶段——对于从未构建过一门生意的初次创业者来说，这可能是一个挑战。Claude 能通过构建一份"现阶段只有你该做的事"的清单来提供帮助，这可能包括产品叙事决策、董事会关系、企业交易和创始人对创始人的对话之类的事项。不在那份清单上的任何事，都是委派或 Claude Cowork 自动化的候选项。

练习

用 Claude 产出一张你当前运营层的瓶颈地图：每一个目前经由你路由的工作流、决策和审批。现在，请 Claude 推演当你有一周不在时,每一个会发生什么。停滞的那些工作流，正是你仍然亲力亲为到足以妨碍进展的地方。这些如何映射到你与 Claude 一起制作的创始人优先级与职责清单？接下来，是时候压力测试你已经构建的系统是否真的准备好随你的业务增长而扩展了。

把技术运营扩展为企业级基础设施

随着你扩展，买家需要确信你的产品和你的组织能作为长期基础设施被信任。技术工作仍然一如既往地在代码库内部进行，但现在也有围绕代码库的技术工作要处理。第一步是把机构知识转化为一个能扩展的系统。用 Claude 来起草和维护企业采购期望看到的书面基础设施，包括产品文档、支持操作手册和 SLA。与此同时，指挥 Claude Code 对照企业合同要求的特定可靠性和安全标准来审计和加固代码库，并构建出基于 Discord 的社区支持从来不必提供的技术支持基础设施：日志、监控、事件响应工具，以及让 SLA 真正可执行的可观测性层。然后 Claude Cowork 运行企业支持本身的运营层：工单路由、升级工作流、由产品变更触发的文档更新、续约追踪，以及企业客户成功所依赖的报告节奏。这三者合在一起，给一个小团队以一家大得多的组织的支持姿态——这恰恰是签署一份多年企业合同所要求你展示的。

练习

挑出你最苛刻的三个潜在客户，或识别出三个你很想签下的理想客户。请 Claude 产出一份缺口分析：在签署一份多年合同之前，这些客户的企业采购团队会期望看到什么文档、SLA 和支持基础设施，以及你目前在哪里有所欠缺？用这份输出来把跨 Claude Code 和 Claude Cowork 的技术与文档工作排序。

构建一个真正的 GTM 职能

创始人的拼劲把你带到了这一步，但扩展你的创业公司需要创建并实施一个真正的市场进入策略。AI 能帮你构建，然后运行那台完整的 GTM 引擎。Claude 能协助从零构建基础性 GTM 资源：市场细分、信息传达架构、分析师关系策略、销售操作手册，以及一旦你开始与公众投资者、企业买家和华尔街分析师打交道就变得重要的、面向投资者的指标叙事。这些受众各有自己的词汇，并以自己的标准来评估你；Claude 的工作是把你产品的价值主张翻译成一种对每个受众细分都相关的产品营销方式。如今，Claude Cowork 能成为你的战术执行层：内容管道、外呼序列、分析师简报会事务、新闻室和 PR 节奏、CRM 卫生、管道报告，以及把 GTM 策略变成实际商业动作的众多循环往复周期。在 GTM 动作需要产品营销基础设施的地方——交互式演示环境、集成文档、沙箱租户、API 参考、技术单页文档——Claude Code 能为你构建它。买家期望在技术上评估你的产品，而在规模化阶段，一段 Loom 视频和一份销售演示文稿不再够用。这也是让你的 GTM 动作能异步运行的基础设施：一个构建良好的演示环境会在你身处董事会会议时成交。

把领域专长和机构知识转化为 AI 上下文

许多超精益创业公司的创始人，正在为他们亲身经历或第一手观察到的某个特定行业的真实世界问题构建高度具体的应用或工具。智能体 AI 如今让从未写过一行代码的创始人有可能运用他们的领域专长来构建解决复杂问题的产品。Claude、Claude Code 和 Claude Cowork 各自在把创始人知识转化为复利累加的产品特异性中扮演一个角色。

用 Claude 来捕获、组织和提炼创始人知识，把领域专长放在产品能触及的地方。通过延展的对话、项目和记忆，创始人可以把他们所知的一切——行业术语、监管陷阱、边缘情况、挫折、为什么对这个问题的显而易见的答案行不通的原因——分享进一个结构化、可搜索的上下文。然后，技能（Skills）能把循环往复的工作流（例如"我如何审计一份商业租约"、"我如何分类整理一份患者接诊表")编码成 Claude 每次都以同样方式运行的可复用例程。数月之后，这变成一个专有的知识基底，没有任何通用 AI 能够匹敌。

用 Claude 把你的领域知识外部化，对于把行业特定的边缘情况编码进你的产品变得无比宝贵：例如，一个通用的 AI 医疗账单工具在 340B 药品计划的理赔上会失灵，但你的工具有专门针对它们的逻辑。Claude Code 帮你把你所在领域其他专业人士经历的常见挫折,翻译成验证逻辑、提示词优化，或与一个你的竞争对手闻所未闻的小众行业系统的 MCP 集成。结果是，你的应用或工具的深度和广度都持续复利累加，以一种竞争对手根本无法复制的方式。

练习

识别出一个通用竞争对手在你的垂直领域里肯定会搞错的边缘情况。与 Claude Code 一起，基于一个你实际见过的场景，为它构建一个专门的测试用例（不是单元测试）。每当一个类似的边缘情况浮现时，就把它加进去。你的测试套件就成为你护城河的一张地图。

把累积的用户数据复利累加成一项可防御的优势

随着用户与你的产品互动，他们生成行为信号（即他们接受哪些输出、拒绝哪些），这些信号为产品路线图提供信息。随着时间推移，你将了解你特定用户群的具体模式、偏好和边缘情况。这就是我们所说的复利累加的价值：每一次改进都让产品更有用，从而驱动更多使用，从而创造更多反馈，从而驱动更多改进。这份数据是时间锁定的、上下文特定的，且对一个模仿者来说不可能重建：你根本买不到那数千名一直在你产品内打磨工作流的用户的行为指纹。

练习

给 Claude 喂一份你产品交互数据的摘要：你一直在收集什么、收集了多久，以及你对用户随时间如何与你产品互动的了解。请它识别那份数据中三个最高信号的行为模式，并设计一个把持续使用转化为系统化模型改进的反馈回路。然后请它帮你起草一份一页的护城河叙事，为产品营销提供信息：你的数据飞轮如何运转、它已经旋转了多久，以及为什么一个今天才起步的资源充裕的竞争对手无法在两年内复制它。

创造工作流锁定

复利累加的数据网络效应让你的产品更难复制，而用户工作流锁定让你的产品更难离开。用户在他们的日常运营中运行你产品的时间越长，它就越深地嵌入他们实际工作的方式。他们在它之上构建了自动化、培训人员使用它，并把它连接到他们的数据源和其他工具。他们开发的提示词、他们打磨的工作流、他们标准化的输出，全都围绕你产品做什么、以及它如何做来塑造。到了这个地步，切换就从一个产品决策变成了一个全规模的运营项目。

创造工作流锁定的第一步，是请 Claude 按集成深度绘制你当前的客户群。对于每个客户细分，识别他们在你产品之上构建了什么工作流、他们依赖哪些集成。这显示出你的产品在哪里粘住，以及它需要往哪里走得更深。你提供的集成越多，客户构建依赖你产品的工作流的表面积就越大。Claude Code 帮你快速搭起与你目标用户依赖的数据管道、项目管理工具和其他系统的原生集成。Claude Code 还能构建那些让客户不只是使用你的产品、而是在它之上构建的 API、webhook 和 SDK——这是最深形式的锁定。

练习

请 Claude 帮你为你的前十大客户构建一份工作流集成审计。对于每一个，记录他们构建的自动化、他们依赖的集成、贯穿你产品运行的团队工作流，以及你对他们切换成本的估计。然后请 Claude 识别这一群体中的模式：哪些类型的集成为你的具体产品创造最深的锁定，以及你能构建或启用什么,来为目前还停留在表面的客户加深集成。

第七章

工作相同，规则全新

在 AI 时代，创始人的工作并没有改变：找到一个真实的问题，构建一个解决它的东西，并把它扩展成一家重要的公司。改变的是抵达那里的路径。横跨四个阶段——想法、MVP、发布和规模化——AI 把数个季度压缩成数周。

过去要花数月的验证周期，现在只需数个下午。一个能用的原型不再需要一位拥有正确技术栈的联合创始人；它需要一个清晰的问题，以及与一个编程智能体的几次专注会话。发布就绪从一场发布前的手忙脚乱，压缩成一条持续的工作流。而在规模化阶段，那种过去迫使早期招聘的人陷入救火角色的运营重负，越来越能被交接给 AI，把你团队的注意力解放出来,投向那些成为你护城河的判断决策。

瓶颈不再是你能构建什么，而是你选择构建什么。
